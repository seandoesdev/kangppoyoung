# 청크 병합 스테이지 — LLM 기반 청크 품질 개선 (설계)

> 상위: [`plan_overview.md`](plan_overview.md) P4 (+P1·P3 연계). 구현 전 설계.
> 기준선: [`../eval/results-2026-06-20-baseline.md`](../eval/results-2026-06-20-baseline.md).

PDF→XML 파이프라인의 청킹이 끝난 뒤, **검색에 불리한 파편 청크**를 LLM으로 병합해
**"업무적으로 유효한 가장 작은 의미 단위"**(예: 절차면 단계 전부가 한 청크)로 만든다.
LLM은 (1) 다음 청크와 합칠지 **판단**, (2) 병합 결과가 제대로 된 단위인지 **검증** 두 역할만 한다.

## 1. 목표 · 성공 기준 · 비목표

- **목표**: 청크 = 업무적으로 유효한 가장 작은 완결 단위. 절차/목록/조건이 한 청크에 통째로.
- **성공 기준**(eval 재채점): 절차/목록형(3·4번) 점수 상승, evidence 단위 완결성↑,
  런타임 보정(부모-문서 확장·2-hop·문서 균형) 의존도↓. 회귀 없음.
- **비목표**: 청크 **분할**(merge-only), 이미지 미추출 콘텐츠 복구.

## 2. 배경 · 근본 원인

현 청킹은 절차 8단계를 ~30자 8개 청크로 쪼개는 등 **과소청킹**이라, 검색이 완결 단위를 못 회수한다.
지금까지의 런타임 보정은 파편을 사후 봉합하는 반창고였다. 본 스테이지는 **소스에서 단위를 바로잡는다.**

## 3. 설계 개요 — 파이프라인 1 스테이지

- **위치**: `build`(청크 형성, reading order·heading_path 보유) **직후** ~ `relations`/`embedding`/`serialize` **직전**.
  병합 후 chunk_id·관계·embedding_text를 정상 흐름대로 재계산한다.
- **무손실**: 병합 = **기존 청크 span을 순서대로 그대로 연결**(재작성 0) → `verify`의 보존율(≥0.99)·round-trip 자동 유지.
- **결정론**: 병합/검증 판단을 **content-hash로 캐시** → 재실행 시 동일 출력(test_determinism 유지).
- **오프라인 기본**: 선택적·키 게이트(기존 `providers` 패턴). 키 없으면 **무병합**(원본 청크 유지).

## 4. Pass 1 — 경계 검출 / 병합 판단 (보강 1·2)

reading order 청크 시퀀스를 순차 스캔하며 **"i와 i+1 사이에 단위 경계가 있는가?"**(boundary detection)를 판단한다.

- **보강 1 — 컨텍스트 윈도**: 판단 입력 = *현재까지 병합한 그룹 요약* + *heading_path* + *다음 청크*.
  → 두 청크만 보지 말고 **"다음 청크가 이 단위를 계속 잇는가?"** 로 묻는다(단위가 어디서 끝나는지 알 수 있음).
- **보강 2 — 경계·크기 차단(과병합 방지)**:
  - **heading 경계는 기본 분리**(새 소제목 = 새 단위).
  - 누적 크기가 `max_chunk_chars`(기존 800) 초과 시 **STOP**.
  - 단위가 완결되면 STOP(다음 단위로 넘어가 욕심 병합 금지).
- **출력**: 병합 그룹(연속 청크 묶음).
- **비용**: heading 섹션 단위로 **배치 호출**(한 번에 여러 경계 판단).

## 5. Pass 2 — 검증 (보강 3)

각 병합 청크가 "제대로 된 단위"인지 LLM이 검증한다.

- **기준**: 단일 단위 · **완결**(절차면 단계 전부, 끊긴 문장/중간 누락 없음) · 크기 적정.
- **실패 시 동작**: 처음엔 **비차단 플래그**(`review_required` + 사유, 파이프라인 기존 패턴). 보정 재병합/분할은 후속.
- 병합으로 못 고치는 **과대·혼합 청크**도 여기서 플래그만 남긴다.

> 프레이밍 팁: "다음과 합칠까?"보다 "사이에 경계가 있는가?"가 프롬프트가 더 깔끔(결과 동일).
> Pass 1(생성) + Pass 2(검증)은 곧 generate-then-verify 자기검증.

## 6. 무손실 · 결정론 · 오프라인 (상세)

- **무손실**: 병합 텍스트 = 원본 청크 텍스트의 순서 연결. `embedding_text`는 병합 텍스트에서 재파생.
  `verify` 보존율·round-trip을 **그대로 통과해야 함**(통과 못 하면 병합 적용 안 함).
- **결정론**: 캐시 키 = `hash(그룹 컨텍스트 + 다음 청크)`. `providers`처럼 Offline/OpenAI MergeProvider + 캐시.
- **오프라인**: `config` 플래그 + 키 없으면 `OfflineMergeProvider` = 항상 "경계 있음"(원본 유지).

## 7. 범위 한계 (정직히)

- **merge-only**: 이미 과대·혼합된 청크는 **분할 못 함**(분할은 무손실 보장이 깨지기 쉬움) → 플래그만.
  현 문제는 대부분 파편화라 병합만으로 큰 효과.
- **이미지/순서도 절차**(오프라인=플레이스홀더)는 텍스트가 없어 병합으로 복구 불가 — 별개 과제.

## 8. 파이프라인 변경 · downstream

- **신규**: `pipeline/merge.py`(스테이지), `providers`에 `MergeProvider`(Offline/OpenAI), `build`/`cli` 연결, `config` 설정.
- 병합 후 `relations.py`(prev/next/parent 재계산), `embedding.py`(재파생), `serialize`, `verify` 정상 흐름.
- **chunk_id 재계산**(병합 내용 해시) → 관계 무결성 유지.

## 9. 프롬프트 스케치

- **병합 판단**: "정책자금 문서의 연속 청크다. [현재 단위 요약]. 다음 청크가 같은 '업무적으로 유효한 단위'를
  잇는가? 새 소제목/주제 전환이면 경계=true. boolean만."
- **검증**: "이 병합 청크가 단일·완결 업무 단위인가? 절차면 단계 누락/순서, 끊긴 문장 여부 점검. valid + 사유."
- 둘 다 구조화 출력(boolean / record).

## 10. 리스크 · 완화

| 리스크 | 완화 |
|---|---|
| 과병합(욕심) | heading 경계·max 크기 STOP + Pass 2 검증 |
| 단위 경계 오판 | 컨텍스트 윈도(그룹+heading), 검증 자기점검 |
| granularity 일관성 | 프롬프트에 기준·예시 명시 |
| 결정론 깨짐 | content-hash 캐시 |
| 보존율 위반 | 재작성 금지(연결만) + verify 게이트(미통과 시 미적용) |
| 재임베딩 비용 | 일회성, offline 기본은 무영향 |

## 11. 테스트

- **단위**(`merge.py`): 연속 절차 청크 → 1청크 병합. heading 경계서 분리. max 크기서 STOP. 캐시 결정성(동일 입력→동일 출력).
- **파이프라인**: 골든 문서 재청킹 후 `verify` 보존율 ≥0.99·round-trip pass. '다. 융자절차' 섹션이 1청크인지.
- **평가**: `search-eval` 재채점(3·4번·다문서) 베이스라인 대비.

## 12. 운영 · 단계 · 롤백

- **재적재 필요**(청크·임베딩 변경). offline 기본은 변화 없음(opt-in).
- 단계: (1) `merge.py` + Offline no-op + 단위테스트 → (2) OpenAI provider + 캐시 → (3) 골든 재청킹·verify
  → (4) 재적재 + eval 회귀 측정.
- **롤백**: `config`로 스테이지 off → 기존 청킹 그대로.

## 13. 검증 기준

`verify` 보존율·round-trip 무결을 전제로, `search-eval` 재채점에서 절차/목록형(3·4번) 상승과
부모-문서 확장 의존 감소를 확인한다. 노이즈 완화를 위해 항목별 N회 평균 채점 병행 권장.
