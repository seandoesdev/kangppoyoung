# 구현 계획 (Implementation Plan)

앞으로 구현할 작업을 우선순위로 정리한다. 시스템은 이미 end-to-end로 동작
(PDF→RAG 청크→벡터 검색·gpt-4o 답변 합성 + 공고 버전관리·랭킹·온보딩)하며,
본 문서는 **현재 베이스라인 대비 개선 작업**에 집중한다.

- 검색 품질 기준선: [`../eval/results-2026-06-20-baseline.md`](../eval/results-2026-06-20-baseline.md)
  — 평균 53.8/100, 정답 3·부분 4·오답 3, **2출처 실제 조합 1/10**.
- 모든 검색 개선은 [`../eval/search-eval.md`](../eval/search-eval.md) 세트로 재채점해 베이스라인과 회귀 비교한다.

---

## P1. 다문서 커버리지 보장 (검색 품질, 최우선)

> 세부 구현 설계: [`p1_impl_plan.md`](p1_impl_plan.md)

- **문제**: 답이 2개 이상 문서를 합쳐야 완성되는 질의에서 한 문서(주로 변경공고)에 쏠림.
  평가 세트가 2출처 조합을 요구하는데 10문항 중 1문항만 성공(5번).
- **원인**: 섹션 다양성 캡은 *한 문서 내* 섹션 쏠림만 막고, *서로 다른 문서*의 상보 정보 회수를 보장하지 않음.
- **접근**:
  - retrieval 후보가 여러 문서에 걸치도록 보장(문서당 최소 N개 확보 / 문서 단위 다양성).
  - 2-hop `RetrievalRefiner`를 확장: "다른 문서의 상보 정보가 빠졌는지"까지 점검하고, 부족하면
    해당 문서를 겨냥한 followUpQuery 생성.
- **검증**: eval 2·4·6·9번 점수 상승, 2출처 조합률 ≥ 절반.

## P2. refiner 게이트·합성 범위 확대

- **문제**: 예/아니오·구분/목록형 질의(eval 8번 "예", 4번 "근거 없음")에서 과도한 축약 또는 거짓 근거없음.
- **접근**: refiner 가동 게이트를 `절차/목록`에서 `예아니오·정의·자격·구분` 등으로 확대.
  합성 프롬프트가 결론 + 근거 설명(구체 항목·품목)을 함께 제시하도록 강화.

## P3. 수치 충실도

- **문제**: %p·한도 등 정량 정보 오기/누락(eval 6번 우대 0.4%p→0.3%p 오기, 9번 한도 60→100억 누락).
- **접근**: 합성 프롬프트에 "근거의 수치·비율·한도는 원문 그대로 인용" 강화. 수치 포함 청크 우선 회수.

## P4. 청크 품질 — 업무 단위 재청킹 (근본 개선)

> 세부 설계: [`chunk_merge_impl_plan.md`](chunk_merge_impl_plan.md)

- **문제**: 청킹이 절차·목록을 과소청킹(8단계 → 8조각)해 검색이 완결 단위를 못 회수.
  eval 3·4번 등 다수 실패의 **뿌리**이며, P1·2-hop 등 런타임 보정이 보상하려던 근본 원인.
- **접근(근본)**: 파이프라인에 **LLM 병합 스테이지** 1개 추가 — 청크를 '업무적으로 유효한 가장 작은
  단위'(절차면 단계 전부 = 한 청크)로 병합. LLM은 (1) 다음 청크와 합칠지 판단 + (2) 결과 검증만 수행.
  **무손실(연결만)·결정론(캐시)·오프라인(게이트)** 유지. 재청킹 + 재적재 필요.
- **효과**: 절차/목록 정확도(P4) + 다문서 회수 용이(P1) + 단위 완결로 수치 누락↓(P3)을 **함께** 개선.
  한 번 데이터를 제대로 만들어 런타임 보정 의존을 낮춘다.

## P5. 근거–답변 정렬

- **문제**: 답변이 실제로 쓴 근거가 "근거 조항" 카드에 안 보임(카드 = 코사인 상위 5건, 답변 출처와 불일치).
- **접근**: 합성 LLM이 사용한 근거를 함께 반환하게 해 카드를 답변 출처와 정렬(검색 상위로 폴백).

## P6. 프론트엔드 — 목 데이터 → 실 백엔드 연동

- **현황**: 통합 검색만 라이브. **랭킹·온보딩·정책공고 페이지는 아직 `frontend/src/data/mock.ts`** 사용.
- **접근**: 각 페이지를 백엔드 API(`/rankings`, `/onboarding`, `/notices/...`)에 연동.

## 운영·기타

- 결정성: 합성 temperature 0 고정으로 실행별 변동 완화.
- 비용·지연: 2-hop 절차 질의 ~10–18s. 게이트 정교화·결과 캐싱 검토.
- 평가 자동화: LLM-judge 기반 자동 채점 스크립트화(세트 회귀 비교 상시화).
- 테스트: 백엔드 통합테스트는 Docker(Testcontainers) 필요. 검색 provider는 테스트에서 hash/offline로 핀됨.
