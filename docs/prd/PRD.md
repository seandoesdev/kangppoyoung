# PRD — 정책자금 지원 업무 플랫폼

> 제품 요구사항 문서(Product Requirements Document)
> 관련 문서: [User Flow](../user_flow/user_flow.md)
> 버전: v0.1 (프로토타입 검증 단계) · 최종 수정: 2026-06-06

---

## 1. 개요 (Overview)

정책자금 지원 업무담당자가 방대한 **규정·지침·절차**를 빠르게 찾고, 변경 사항을
최신 상태로 유지하며, 신규입사자가 무엇부터 학습해야 하는지 데이터 기반으로
안내받을 수 있는 **사내 업무 지원 플랫폼**이다.

핵심 가치는 "흩어진 문서를 사람이 일일이 찾는" 비효율을 제거하고,
**단일 진실 문서(Single Source of Truth)** 위에서 검색·갱신·학습이 선순환하도록
만드는 것이다.

---

## 2. 문제 정의 (Problem Statement)

| 문제 | 현재의 고통 |
| --- | --- |
| 문서가 흩어져 있다 | 규정·지침·절차를 하나씩 찾는 데 시간이 소모된다 |
| 실시간 응대가 어렵다 | 전화 민원 응대 중 즉석으로 근거를 찾기 어렵다 |
| 변경 추적이 누락된다 | 절차가 개정되어도 갱신 누락·구버전 혼용이 발생한다 |
| 무엇이 중요한지 모른다 | 자주 묻는 내용이 무엇인지 파악되지 않는다 |
| 온보딩 기준이 없다 | 신규입사자가 무엇부터 봐야 할지 알 수 없다 |

---

## 3. 목표 (Goals) & 비목표 (Non-Goals)

### 3.1 목표
- 자연어 질의 한 번으로 근거 조항까지 포함한 답변을 제공한다.
- 중복·상충 문서를 명확한 규칙으로 처리하여 신뢰할 수 있는 답을 준다.
- 개정 문서를 단일 진실 문서의 새 버전으로 누적하고 변경 사항을 시각화한다.
- 실제 검색·조회 데이터를 학습 우선순위로 환산해 온보딩을 자동화한다.

### 3.2 비목표 (현 단계 제외)
- 별도의 AI 추천 로직(검색 데이터와 무관한 임의 추천)은 만들지 않는다.
- 외부 민원인(고객) 직접 사용 채널은 범위 밖이다(업무담당자 내부용).
- 결재·전자문서 워크플로우 통합은 추후 검토한다.

## 5. 핵심 기능 요구사항 (Functional Requirements)

### 5.1 UC-1 — 통합 검색 (자연어 질의 + 민원 응대)
**경로:** `/`

일반 질의와 전화 민원 응대를 하나의 화면으로 통합한다. 두 경우 모두 결국
규정·지침·절차를 찾는 동일한 목적이기 때문이다.

| ID | 요구사항 | 우선순위 |
| --- | --- | --- |
| FR-1.1 | 자연어로 규정·지침·절차·민원 내용을 질의할 수 있다 | P0 |
| FR-1.2 | 답변에는 **항상 출처(문서명·조항)**를 명시한다 | P0 |
| FR-1.3 | **중복 절차**는 요약 1건 + 출처로 표시한다 | P0 |
| FR-1.4 | **상충 절차**는 임의 통합 없이 원문 그대로 병렬 + 출처로 표시한다 | P0 |
| FR-1.5 | 예시 질문을 사용자가 **최대 5개**까지 추가/삭제, 클릭 시 즉시 검색 | P1 |
| FR-1.6 | 질문·답변을 DB에 저장하여 검색 기록으로 누적·재조회한다 | P1 |

**수용 기준(AC)**
- 중복은 절대 둘 다 나열하지 않고 요약 1건으로 합친다(출처는 모두 표기).
- 상충은 절대 임의 통합하지 않고 원문을 병렬 표시한다.

### 5.2 UC-3 — 정책 자금 공고 (개정본 등록 & 버전 비교)
**경로:** `/notice/regulation`(공고), `/notice/reference`(참고자료)

각 문서는 **단일 진실 문서**로 유지되고, 개정 시 새 버전이 누적된다.

| ID | 요구사항 | 우선순위 |
| --- | --- | --- |
| FR-3.1 | 메뉴 구조: 정책 자금 공고 > 공고 / 참고자료 | P0 |
| FR-3.2 | 버전 드랍박스는 **날짜 내림차순(최신 우선)**으로 정렬한다 | P0 |
| FR-3.3 | 개정본 등록은 3단계 마법사: ①PDF 업로드 → ②전처리 → ③검토·승인 | P0 |
| FR-3.4 | 전처리(텍스트 추출·표/도표 인식·정규화)는 자동 수행한다 | P0 |
| FR-3.5 | 검토 화면은 좌(이전, 읽기전용) ↔ 우(갱신본, 편집가능) 병렬 표시 | P0 |
| FR-3.6 | 등록 확정은 **사용자 승인 후**, 시행일 입력 후에만 이루어진다 | P0 |
| FR-3.7 | 변경 사항은 **바로 전 버전 대비 블록 단위 diff**로 표시한다 | P0 |
| FR-3.8 | diff 표기: 추가=초록, 삭제=빨강 (텍스트·이미지 블록 모두) | P0 |

**수용 기준(AC)**
- 전처리 자동화는 등록을 자동 확정하지 않는다(반드시 사용자 승인 게이트).
- diff는 LCS 기반 블록 비교로 추가/삭제/동일을 구분한다.

### 5.3 UC-4 — 유사 질문 카테고리·랭킹
**경로:** `/ranking` (메뉴명: 질문 분석)

| ID | 요구사항 | 우선순위 |
| --- | --- | --- |
| FR-4.1 | 사용자가 집계 기간(예: 최근 7일/30일)을 선택할 수 있다 | P0 |
| FR-4.2 | 저장된 질의·답변에서 유사 질문을 카테고리화한다 | P0 |
| FR-4.3 | 빈도순(검색/조회 횟수) 랭킹을 산출·표시한다 | P0 |
| FR-4.4 | 각 카테고리의 핵심 근거 문서·조항을 함께 보여준다 | P1 |
| FR-4.5 | 추세(상승/하락/유지) 지표를 표시한다 | P2 |

### 5.4 UC-5 — 신규입사자 온보딩 (UC-4 랭킹 기반)
**경로:** `/onboarding` (메뉴명: 온보딩 가이드)

| ID | 요구사항 | 우선순위 |
| --- | --- | --- |
| FR-5.1 | 학습 우선순위는 **UC-4 랭킹 데이터**에서 도출한다(임의 추천 금지) | P0 |
| FR-5.2 | "많이 보고·많이 검색한" 순서를 학습 순서로 환산한다 | P0 |
| FR-5.3 | 각 학습 항목에 **선정 근거(검색/조회 N회)**를 표시한다 | P0 |
| FR-5.4 | 신규입사자의 질의·조회도 DB에 누적되어 랭킹에 재반영(선순환) | P1 |
| FR-5.5 | 기간 변화로 랭킹이 바뀌면 온보딩 우선순위도 자동 최신화 | P1 |

## 6. 데이터 흐름 (Data Flow)

```
[원문 문서(규정/지침/절차)]
        │  업로드 + 전처리 파이프라인(그림·도표 → 텍스트)
        ▼
[단일 진실 문서 저장소]  ◄── UC-3 개정 시 이 문서만 갱신(새 버전 누적)
        │  검색·인덱싱
        ▼
[질의 응답 엔진]  ── 출처 명시, 중복=요약 / 상충=원문 병렬
        │
        ├─► [사용자 답변]
        ▼
[질의·답변 DB]  ── UC-4 기간별 유사 질문 카테고리·랭킹
        ▼
[UC-5 온보딩]  ── 랭킹(많이 보고·많이 검색한 순) = 학습 우선순위
```

핵심 설계 원칙: **UC-4 랭킹이 UC-5 온보딩의 유일한 데이터 소스**이며,
사용자의 모든 질의가 다시 랭킹으로 환류되어 선순환을 형성한다.

## 7. 데이터 모델 (핵심 엔티티)

| 엔티티 | 핵심 필드 | 설명 |
| --- | --- | --- |
| PolicyDocument | id, title, type(규정/지침/절차), updatedAt, isSingleSource | 단일 진실 문서 |
| Article | docId, docTitle, docType, articleNo, text | 조항 단위 근거 |
| SearchResult | query, answer, evidence[], duplicateSummary?, conflicts[] | 검색 응답 |
| RankingItem | rank, category, questionExample, searchCount, viewCount, trend, relatedArticles[] | 랭킹 항목 |
| NoticeCategory | key(regulation/reference), label, docTitle, versions[] | 공고 문서 |
| NoticeVersion | version, date, blocks[] | 문서 버전 |
| ContentBlock | text \| image | 본문 블록(텍스트/이미지) |
| DiffBlock | type(same/add/del), block | 버전 간 변경 단위 |

> 프로토타입 단계의 타입 정의는 `frontend/src/data/mock.ts`에 구현되어 있다.

## 8. 비기능 요구사항 (Non-Functional)

- **정확성:** 답변은 항상 근거 조항을 동반한다(근거 없는 답 금지).
- **무결성:** 단일 진실 문서 원칙 — 동일 문서는 버전으로만 누적·갱신한다.
- **추적성:** 모든 개정은 시행일·버전·diff로 추적 가능해야 한다.
- **안전성:** 전처리 자동화 결과는 사용자 승인 게이트를 통과해야 등록된다.
- **사용성:** 민원 응대 중 즉시 입력·재조회 가능한 빠른 반응성.
- **확장성:** 검색·랭킹·온보딩이 동일 데이터 위에서 일관 동작.
