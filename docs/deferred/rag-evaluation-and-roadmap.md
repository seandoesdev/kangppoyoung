# RAG 평가 종합 및 향후 방향

> 통합 검색(`POST /api/v1/search`, 벡터 코사인 + OpenAI 합성)의 현재 품질과 다음 작업 방향을 정리한다.
> 측정 기록 원본: [`../eval/`](../eval/) (baseline / p1 / merge / avg10 / temp0-experiment / retrieval).
> 평가 세트: [`../eval/search-eval.md`](../eval/search-eval.md) (10문항, 2문서 이상 조합 필요하게 설계).

## 0. TL;DR

- **현재 신뢰 성능: 72.1 / 100** (10회 평균). 100개 답변 중 **정답 38% · 부분정답 49% · 오답 13%**.
- **가장 큰 교훈**: 1회 측정은 합성 LLM 비결정성으로 **±15~40점 노이즈**라 신뢰 불가. 같은 corpus가
  1회=56.4, 10회 평균=70.7. **반드시 10회 평균으로 판단**해야 한다(하니스 구축 완료).
- **강점**: 절차·목적·자금연결 등 **구조형 질의**(병합 효과). **약점**: 품목표 회수(Q8)·수치 디테일.
- **다음 한 수**: 하이브리드 검색(lexical+vector) — Q8·Q6·Q9를 한 번에 개선할 가장 큰 레버리지.

## 1. 평가 방법론

| 시기 | 방법 | 신뢰도 |
|------|------|------|
| 초반 | 항목당 **1회** 실행 채점 | ❌ ±15~40점 노이즈 |
| 현재 | 항목당 **10회** 실행 평균 (`eval/collect.py` + LLM-judge 워크플로우) | ✅ mean±std |

- **하니스**: `eval/eval_set.json`(골든 단일 소스) → `eval/collect.py`(질문당 N회 `/search`, 동시 2 + 503 백오프)
  → 채점 워크플로우(항목당 1 에이전트가 N답변 채점) → 항목 mean±std 집계.
- **노이즈 출처**: 검색은 결정적이나 합성(gpt-4o)이 비결정적. + 질의 분석기(searchTerms)도 비결정 → 검색 후보 변동.

## 3. 적용된 개선과 결과

| 개선 | 내용 | 효과 |
|------|------|------|
| **청크 병합** | 파이프라인에 LLM 경계판단 스테이지 — 절차/목록을 업무 단위로 병합(무손실·결정론) | 절차형(Q3) 55→**96** 등. 구조형 질의의 토대. |
| **문서 균형 검색(P1)** | 관련도 게이트 통과 문서당 최소 쿼터 → 다문서 회수 보장 | evidence 2문서 **10/10**. 다출처 조합. |
| **가중 교차 병합(2:1)** | 원 질의(결정적) + 분석기 질의를 2:1 교차 → 핵심 청크 회수 안정 | 70.7→**72.1**. Q4 "근거없음" 3/10→1/10, Q6·Q5·Q2 recall↑. |
| **합성 프롬프트(Q8)** | 예/아니오형도 근거 설명 강제 | 품목표 회수 시 풀어 답함(但 회수가 병목). |
| **10회 평균 하니스** | 신뢰 측정 인프라 | 모든 튜닝의 판단 기준. |

**되돌린 시도(음성 결과도 기록):**
- **temperature 0**: 결정론화 시도 → 평균 −1.2, Q4 미해결(OpenAI 잔여 비결정성) + Q8 회귀(−18, 단답 고정). 되돌림.
- **교차 1:1**(원 질의 동등 비중): Q4↑지만 분석기 의존 질의(Q9) 상쇄 → 보합. 2:1로 조정해 해결.

## 4. 진단된 약점 (근본 원인)

1. **🔴 Q8 품목표 미회수** — 참고자료의 중점지원 품목표가 **table-row 청크로 흩어져** 있고, 그
   `embedding_text`가 질의("스마트팩토리…중점지원")와 어휘 매칭이 약하다. 근거가 없어 합성이 단답 "예"(8/10).
2. **🟠 Q4 성장단계 변동** — 병합으로 funds 청크가 생겼고 교차병합으로 "근거없음"은 줄었으나, 성장단계
   라벨(창업기/성장기/재도약기/전주기) 구분이 표본마다 들쭉날쭉. funds 청크 회수가 분석기/원질의 비중에 민감.
3. **🟡 수치·예외 충실도** — Q6 우대 0.4%p, Q9 평가면제·수시접수 등 *구체 수치·예외*를 합성이 일부 누락
   (원문엔 존재). 일반론으로 뭉개는 경향.
4. **🟡 비결정성** — 합성·질의분석 LLM 변동으로 같은 질의도 답이 출렁(Q2 std 17, Q4 std 28).
5. **무정보 청크 혼입** — 표지·"< 목 차 >" 등이 상위 후보에 올라와 슬롯 낭비.

## 5. 향후 방향 (우선순위 로드맵)

> 우선순위 = (영향 × 확신) / 노력. 각 변경은 **10회 평균 하니스로 검증**.

### P1 — 하이브리드 검색 (lexical + vector) 〔영향 최대〕
현재 순수 벡터(코사인)라 **정확한 품목명·수치·고유명사 매칭이 약하다**. BM25/키워드 검색을 병합하면
Q8(품목명)·Q6/Q9(수치) 정확 매칭이 살아난다. **Q8·Q6·Q9를 한 번에 개선할 단일 최대 레버리지.**
- 구현: MySQL FULLTEXT(ngram) 또는 경량 BM25 인덱스 → 벡터 후보와 RRF(Reciprocal Rank Fusion) 병합.

### P2 — Q8 품목표 회수 〔높은 확신, 중간 노력〕
P1로 상당 부분 풀리지만 추가로:
- 청크 병합 스테이지를 **table-row까지 확장** — 한 품목표(테마·분야·품목)를 한 의미 단위로 병합.
- table-row `embedding_text`에 **상위 분야·테마 컨텍스트** 포함(현재 row 값 위주).

### P3 — 수치·예외 충실도 〔합성 영역〕
- 합성 프롬프트 강화: "후보의 수치·비율·한도·예외는 원문 그대로 인용, 누락 금지."
- 수치 포함 청크를 후보에서 가볍게 우대(또는 P6 재랭킹으로 흡수).

### P4 — 무정보 청크 제거 〔저노력 정밀도〕
- 표지·목차·title-only 청크를 **ingest 단계에서 스킵**하거나 검색 시 강등.

### P5 — 재현성 / self-consistency 〔품질·비용 trade-off〕
- 비결정성 완화: 핵심 질의에 **N-sample + 다수결**(self-consistency)로 안정화, 또는 seed 활용.
- 주의: temperature 0 단독은 품질 손해(검증됨) — multi-sample 방향 권장.

### P6 — 재랭킹(re-ranker) 〔정밀도 상한 상향〕
- 벡터로 회수한 후보 90개를 **cross-encoder 또는 LLM 재랭커**로 재정렬 → 정밀도↑.
  (Spring AI 또는 별도 모델. 비용·지연 고려.)

### P7 — 평가 확장 〔측정 신뢰도 강화〕
- 골든셋 **10 → 50+ 확대**(카테고리·난이도 균형, 단일문서/다문서/수치/예아니오 골고루).
- 하니스 원커맨드화 + 회귀 CI 편입. 병합-전 10회 baseline도 1회 측정해 병합 순효과를 엄밀화.

## 6. 설계 원칙 (이번 작업에서 얻은 학습)

1. **측정 신뢰성이 먼저** — 1회 노이즈에 속아 잘못된 결론(회귀 오인)을 내릴 뻔했다. N회 평균이 기본.
2. **청킹이 토대** — 검색·합성보다 근본. 절차 병합 하나로 Q3가 안정 95가 됐다.
3. **검색은 recall↔precision trade-off** — 교차병합 비중(2:1)으로 균형. 한쪽으로 치우치면 다른 항목이 상한다.
4. **합성은 재료가 좋으면 잘한다** — 병목은 대개 **검색(재료 회수)**. Q8이 단답인 건 합성이 아니라 회수 실패.
5. **결정론 ≠ 품질** — temp 0은 변동만 줄이고 품질은 손해. 변동은 측정(N회)·self-consistency로 다룬다.
